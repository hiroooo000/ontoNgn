# ドキュメント整合性チェック (Document Consistency Audit)

以下のステップに従い、ドキュメントの整合性を厳密にチェックし、問題があればレポートしてください。

## Step 1: 実体ファイルと目次の照合 (Index vs File System)
- `list_dir` ツールを用いて `docs/` および `docs/detail_design/` の実際のファイル一覧を取得する。
- `docs/00_document_structure.md` を読み込み、目次として記載されているファイルリストと実際のファイルを比較する。
- **チェック項目**:
  - 実体があるのに目次に記載されていないファイル（孤立ファイル）はないか？
  - 目次に記載されているが実体が存在しないファイル（リンク切れ）はないか？
  - `01_` や `02_` などの先頭の連番に「重複」や「抜け」が発生していないか？

## Step 2: 機能一覧と詳細設計ドキュメントの照合 (Features vs Detail Designs)
- `docs/02_system_design.md` の「機能一覧」および「API仕様」を読み込む。
- **チェック項目**:
  - システム全体設計で定義されている各機能モジュール（01〜xx）に対して、それに対応する詳細設計ドキュメント（`docs/detail_design/xx_...md`）が1対1で漏れなく存在しているか？
  - 処理フロー図に登場するアクター・機能が、機能一覧から欠落していないか？

## Step 3: ステータス管理の網羅性チェック (Status Tracking)
- `docs/zz_development_status.md` を読み込む。
- **チェック項目**:
  - `01_specification.md` や `02_system_design.md` で定義されたすべての主要機能が、「実装済み」か「未実装」のどちらかに漏れなく分類・記載されているか？

## 完了条件 (アーティファクト出力)
上記のStepをすべて実行した結果をまとめたMarkdownアーティファクトファイル（`document_audit_report.md` など）を作成してユーザーに提示してください。
チャット欄に直接長文を出力するのではなく、必ずアーティファクトとして出力すること。
不整合が発見された場合は「修正すべき項目」としてリストアップし、不整合がない場合は「すべてのドキュメント整合性チェックをパスしました」と記載すること。
