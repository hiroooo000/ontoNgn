# Graph Visualization (グラフ可視化機能) 詳細設計書

## 1. 概要
本機能は、Kuzu DB内に蓄積されたオントロジーデータ（ノードおよびエッジ）を視覚的に探索・確認するためのUIおよびAPIを提供する。
大規模なグラフ全体を一度に描画することによるパフォーマンス低下を防ぐため、**「キーワード検索を起点としたサブグラフの表示（アプローチB）」** を採用する。

## 2. アーキテクチャと技術スタック
*   **フロントエンド**: Vue 3 / TypeScript (Viteビルド) / TailwindCSS
    *   旧来のJinja2テンプレートによるHTMLべた書き実装から、保守性とテスト容易性向上のため、純粋なTypeScriptモジュール（APIクライアント、データ整形、Vis.js制御）へロジックを分離・抽象化し、単体テスト(Vitest)可能な構成を採用している。
*   **グラフ描画ライブラリ**: `vis-network` (Vis.js)
    *   採用理由: JSONデータからの物理演算（Force-directed）レイアウトの自動生成、ドラッグ＆ドロップ、ズーム、パンなどの直感的な操作が標準で組み込まれており、シンプルな導入が可能であるため。
*   **バックエンド**: FastAPI (ルーター: `app/interfaces/api/graph.py`)
    *   開発時はVite開発サーバーからプロキシするが、デプロイや本番実行時はフロントエンド資産(`frontend/dist`)をFastAPIの `StaticFiles` 機能でマウントし、`@app.get("/graph")` が直接 `graph.html` を返す構成とすることで、Pythonサーバーの単一プロセス起動のみでアプリケーション全体が動作する設計とする。

## 3. UI/UX設計 (画面レイアウト)

画面は大きく分けて「全画面グラフ表示領域」と、操作機能を集約した「左ペイン」で構成される。

### 3.1 画面レイアウト構成
1.  **全画面グラフ表示領域 (Main Canvas)**
    *   画面の背景全体を占める領域。
    *   `vis-network` のキャンバスとして機能し、ノードとエッジが物理演算を用いて動的に描画される。
    *   ノードクリックイベントを捕捉し、左ペインへ情報を伝達する。
2.  **左ペイン (Sidebar Panel)**
    *   画面左側にフロートする形で配置されるパネル。
    *   トグルボタン（`<`, `>` などのアイコン）により、折り畳み（非表示）と展開が可能。
    *   **検索領域 (Top Section)**:
        *   キーワードを入力するテキストボックスと、検索ホップ数（最大10）の選択プルダウン。
        *   検索実行ボタン。
    *   **検索ヒットリスト領域 (Middle Section)**:
        *   検索キーワードに合致したノードの一覧を表示する。
        *   リストのノードをクリックすると、そのノードを新たな起点としてバックエンドからサブグラフを再取得し、グラフ全体を再描画する。

### 3.2 ユーザー操作フロー
1.  ユーザーが左ペインの検索ボックスにキーワードとホップ数を入力し、検索を実行する。
2.  バックエンドAPIがキーワードに合致するノード（アンカー）を検索し、それらを起点とした指定ホップ数のサブグラフと、アンカーノードのリスト（hits）を取得し、UIへ返却する。
3.  `vis-network` がデータを受け取りグラフを描画し、左ペインに検索ヒットしたノードがリスト表示される。
4.  ユーザーがキャンバス上のノードまたはエッジをクリックすると、その要素の近傍に Tooltip 風のフローティングカードが表示され、プロパティ詳細（JSON等）が確認できる。
5.  ユーザーが左ペインのヒットリストからノードをクリックするか、キャンバス上のノードをダブルクリックすると、そのノードを中心とした指定ホップ数のグラフ再取得（expand）が実行され、再描画される。

## 4. API設計

グラフ可視化データを取得するための専用エンドポイントを新設する。

### GET `/api/v1/graph/search`
*   **概要**: 指定されたキーワードに合致するノードを検索し、そのノードを起点とするサブグラフ（ノード・エッジ群）を取得する。
*   **Query Parameters**:
    *   `q` (string, required): 検索キーワード。
    *   `hops` (integer, optional, default=1): 検索ヒットノードからの展開ホップ数。
    *   `limit` (integer, optional, default=50): 検索でヒットさせるアンカーノードの最大数（フロントエンド保護のため）。
*   **Response (200 OK)**:
    ```json
    {
      "nodes": [
        {
          "id": "node_id_1",
          "label": "EntityLabel",
          "properties": {
            "name": "Target Name",
            "type": "..."
          }
        }
      ],
      "edges": [
        {
          "source_id": "node_id_1",
          "target_id": "node_id_2",
          "relation_type": "depends_on",
          "properties": {}
        }
      ],
      "hits": [
        {
          "id": "node_id_1",
          "label": "EntityLabel",
          "properties": {
            "name": "Target Name"
          }
        }
      ]
    }
    ```

### GET `/api/v1/graph/expand`
*   **概要**: 特定のノードIDを中心とし、指定ホップ数分のサブグラフを再取得する。
*   **Query Parameters**:
    *   `node_id` (string, required): 起点となるノードのID。
    *   `hops` (integer, optional, default=1): 展開するホップ数。
*   **Response (200 OK)**:
    *   `/search` と同様の `nodes`, `edges` を返却する（`hits` は不要）。

## 5. バックエンド処理フロー

`app/interfaces/api/graph.py` 内のAPIルーターから、既存の `IGraphRepository` インターフェース（具象: `KuzuGraphRepository`）を呼び出し、以下の処理を行う。

1.  **アンカーノードの検索** (`/search` のみ):
    *   `repository.search_nodes_by_keywords(keywords=[q], top_k=limit)` を実行し、キーワードに合致する起点のノード群（アンカー）を取得する。これをレスポンスの `hits` にもマッピングする。
2.  **サブグラフの取得**:
    *   取得したアンカーノードのIDリスト（または `/expand` で指定された単一の `node_id`）を引数として、`repository.get_subgraph(anchor_ids=[...], max_hops=hops)` を実行する。
3.  **レスポンスの返却**:
    *   取得結果である `ExtractionResult` をAPIのレスポンススキーマに変換し、フロントエンドへ返却する。

## 6. 将来の拡張性
*   **動的ノード展開機能のバリエーション**: 今回実装する再描画（expand）機能に加え、既存グラフに差分をマージする形式でのドリルダウンへの拡張も容易に行える設計となっている。
*   **表示スタイルの動的変更**: ノードの `label` やプロパティに基づいて、`vis-network` に渡す色やアイコン、サイズを動的に変更する表示ルールを追加可能な構成とする。
