# Graph Visualization (グラフ可視化機能) 詳細設計書

## 1. 概要
本機能は、Kuzu DB内に蓄積されたオントロジーデータ（ノードおよびエッジ）を視覚的に探索・確認するためのUIおよびAPIを提供する。
大規模なグラフ全体を一度に描画することによるパフォーマンス低下を防ぐため、**「キーワード検索を起点としたサブグラフの表示（アプローチB）」** を採用する。

## 2. アーキテクチャと技術スタック
*   **フロントエンド**: HTML / JavaScript (HTMX利用想定) / TailwindCSS (またはプレーンCSS)
*   **グラフ描画ライブラリ**: `vis-network` (Vis.js)
    *   採用理由: JSONデータからの物理演算（Force-directed）レイアウトの自動生成、ドラッグ＆ドロップ、ズーム、パンなどの直感的な操作が標準で組み込まれており、シンプルな導入が可能であるため。
*   **バックエンド**: FastAPI (ルーター: `app/interfaces/api/graph.py`)

## 3. UI/UX設計 (画面レイアウト)

画面は大きく分けて「全画面グラフ表示領域」と、操作機能を集約した「左ペイン」で構成される。

### 3.1 画面レイアウト構成
1.  **全画面グラフ表示領域 (Main Canvas)**
    *   画面の背景全体を占める領域。
    *   `vis-network` のキャンバスとして機能し、ノードとエッジが物理演算を用いて動的に描画される。
    *   ノードクリックイベントを捕捉し、左ペインへ情報を伝達する。
2.  **左ペイン (Sidebar Panel)**
    *   画面左側にフロートする形で配置されるパネル。
    *   トグルボタン（`<`, `>` などのアイコン）により、折り畳み（非表示）と展開が可能。
    *   **検索領域 (Top Section)**:
        *   キーワードを入力するテキストボックス。
        *   検索実行ボタン。
    *   **プロパティ表示領域 (Bottom Section)**:
        *   グラフ上で選択されたノード（またはエッジ）の詳細情報（ID、ラベル、およびJSON形式のプロパティ）を成形して表示する領域。
        *   ノードが未選択の場合は「ノードを選択してください」等のプレースホルダーを表示する。

### 3.2 ユーザー操作フロー
1.  ユーザーが左ペインの検索ボックスにキーワードを入力し、検索を実行する。
2.  バックエンドAPIがキーワードに合致するノード（アンカー）を起点とした指定ホップ数のサブグラフを取得し、UIへ返却する。
3.  `vis-network` がデータを受け取り、グラフを描画する。
4.  ユーザーが描画されたノードをクリックすると、そのノードのメタデータとプロパティ（JSON）が左ペインのプロパティ表示領域に反映される。

## 4. API設計

グラフ可視化データを取得するための専用エンドポイントを新設する。

### GET `/api/v1/graph/search`
*   **概要**: 指定されたキーワードに合致するノードを検索し、そのノードを起点とするサブグラフ（ノード・エッジ群）を取得する。
*   **Query Parameters**:
    *   `q` (string, required): 検索キーワード。
    *   `hops` (integer, optional, default=1): 検索ヒットノードからの展開ホップ数。
    *   `limit` (integer, optional, default=50): 検索でヒットさせるアンカーノードの最大数（フロントエンド保護のため）。
*   **Response (200 OK)**:
    ```json
    {
      "nodes": [
        {
          "id": "node_id_1",
          "label": "EntityLabel",
          "properties": {
            "name": "Target Name",
            "type": "..."
          }
        }
      ],
      "edges": [
        {
          "source_id": "node_id_1",
          "target_id": "node_id_2",
          "relation_type": "depends_on",
          "properties": {}
        }
      ]
    }
    ```

## 5. バックエンド処理フロー

`app/interfaces/api/graph.py` 内のAPIルーターから、既存の `IGraphRepository` インターフェース（具象: `KuzuGraphRepository`）を呼び出し、以下の処理を行う。

1.  **アンカーノードの検索**:
    *   `repository.search_nodes_by_keywords(keywords=[q], top_k=limit)` を実行し、キーワードに合致する起点のノード群（アンカー）を取得する。
2.  **サブグラフの取得**:
    *   取得したアンカーノードのIDリストを引数として、`repository.get_subgraph(anchor_ids=[...], max_hops=hops)` を実行する。
3.  **レスポンスの返却**:
    *   取得結果である `ExtractionResult` をAPIのレスポンススキーマに変換し、フロントエンドへ返却する。

## 6. 将来の拡張性
*   **動的ノード展開機能**: 描画されたノードをダブルクリックした際に、その単一ノードを起点とした `hops=1` の再検索（既存グラフへのマージ描画）を行うことで、逐次的なグラフ探索（ドリルダウン）を可能にする拡張が容易な設計とする。
*   **表示スタイルの動的変更**: ノードの `label` やプロパティに基づいて、`vis-network` に渡す色やアイコン、サイズを動的に変更する表示ルールを追加可能な構成とする。
