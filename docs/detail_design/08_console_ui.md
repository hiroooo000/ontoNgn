# 08. Console UI (Minimal UI) 詳細設計

本システムはバックグラウンド実行エンジンを主軸としており、UIは処理のモニタリングと必要最小限のユーザー介入（スキーマ進化の承認など）を行うためのロジッドレスなプレゼンテーション層として設計されています。

## 1. HTMX + Jinja2 による SPAライクなUI

モダンなJavaScriptフレームワーク（React / Next.js 等）を使用せず、バックエンド（FastAPI）から返却されるHTMLフラグメントを **HTMX** を用いてDOMに直接埋め込むことで、状態の同期を排除しシンプルさを追求しています。

- **`index.html`**: メインのレイアウトとシェルの役割を果たします。
- **`partials/*.html`**: ダッシュボードのパネルや、テーブル行、スキーマ進化の承認ダイアログなど、再利用可能なUIコンポーネント（Jinja2テンプレート）です。

## 2. 実装される主な画面機能

### 2.1 ダッシュボード（ステータス監視）
- **ポーリング**: HTMXの `hx-trigger="every 2s"` などの機能を利用し、サーバーの `/api/v1/documents` から最新のドキュメント処理ステータス（`Uploaded`, `Rendering` など）を定期的に取得して画面を更新します。
- **エラー表示**: `Failed` ステータスになったドキュメントについて、エラーログの展開表示と「再試行 (`Retry`)」ボタンを提供します。

### 2.2 ドキュメント登録パネル
- HTML標準の `<form>` に `hx-post="/api/v1/documents/upload"` を付与し、ファイルのアップロードを非同期で実行します。アップロード完了後、ダッシュボードのリスト部分のみがHTMXによって差し替えられます。

### 2.3 スキーマ進化管理パネル
- ステータスが `Pending_Evolution` になったドキュメントに対して、AI Agentが提案する新規クラス定義（プロパティ、型定義）をプレビュー表示します。
- ユーザーが「承認（Approve）」または「却下（Reject）」ボタンを押下すると、その結果がバックエンドの `/api/v1/schema/candidates/{id}/approve` などへ送信され、処理が再開（パイプライン進行）されます。

## 3. ロジッドレス制約の徹底

フロントエンド側には、ビジネスロジックや状態管理（Redux, Vuex など）を一切持たせません。
データの整合性チェック、スキーマ定義の妥当性検証などはすべてバックエンドの Pydantic および Workflow Orchestrator が担います。
Console UIは「バックエンドの現在のステートをそのまま描き出す」役割に専念します。
