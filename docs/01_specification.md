# ontoNgn (Ontology Engine) 仕様書
Version: 2.0.0

---

## 1. ソリューションのコンセプト

### 1.1 背景と課題
現代のエンタープライズ環境において、組織内に存在する膨大な規定、業務マニュアル、契約書、申請書などの構造化されていないドキュメント（PDF、Word、Excel等）から、ビジネス推進に必要な意味的知識や手続きの流れを整理し、利活用することは極めて困難です。
既存のRAG（Retrieval-Augmented Generation）システムでは、ドキュメントのレイアウト情報（表構造、見出しの包含関係など）が欠落した状態でテキストが分割されるため、文脈の異なる断片が検索され、正確な回答が得られない課題がありました。

また、これらのドキュメントからセマンティックな関係性である「オントロジー（知識グラフ）」を手動で設計・維持するには、多大なコストと専門知識が必要でした。

### 1.2 ontoNgnのアプローチ
**ontoNgn（オントロジー・エンジン）** は、多様なドキュメントからセマンティックな知識表現である「オントロジー（知識グラフ）」を自動的かつ自律的に生成・提供するバックグラウンド実行エンジンです。

1. **マルチモーダルLLM (Vision Model) による構造化テキスト抽出**:
   ドキュメントをページ単位で画像にレンダリングし、Vision Modelを用いることで、表や段落のレイアウト構造を維持した構造化テキスト（Markdown）を抽出します。
2. **LLMを用いたセマンティック関係の自動抽出**:
   抽出テキストから、行政手続きやビジネスフローに含まれるアクター、手続き、必要書類、前提条件などのノードおよび関係（エッジ）をスキーマに従って自動抽出します。
3. **AI駆動のスキーマ進化（Schema Evolution）**:
   ドキュメントによって異なる未知の概念（既存スキーマに定義されていないクラス）が発見された場合、AI Agent（`OntologyEvolutionAgent`）がその重要性と既存スキーマへの適合性を推論し、新規クラスの追加や既存属性へのマッピングを開発者に提案。人間が承認することで、PydanticスキーマおよびOWL/TTLオントロジーを動的に自動生成・コンパイルします。

### 1.3 GraphRAGとの連携
生成された高品質な知識グラフは、APIを介して LlamaIndex や LangChain などの GraphRAG（Graph-based RAG）システムへシームレスにロード・インポートできる構造化JSONや、標準的なTurtle（.ttl）ファイルでリアルタイムに提供されます。

## 2. 機能仕様

### 2.1 機能要件 (Functional Requirements)
1. **ワークフロー制御エンジン (Workflow Orchestrator)**:
   - ドキュメントのアップロードからオントロジー生成までのパイプラインを一元管理し、各処理を非同期APIとして疎結合に呼び出す。
   - **状態管理 (State Management)**: ドキュメントの処理状態（`Uploaded`, `Rendering`, `Extracting_Text`, `Generating_Ontology`, `Pending_Evolution`, `Completed`, `Failed`）を追跡・管理する。
   - **エラーリカバリ**: 各ステップでの処理失敗時にリトライを行う機構や、エラー内容のロギング・通知を行う。
   - **非同期タスク制御**: バックグラウンドキューを利用してタスクを連鎖させ、長時間実行される処理（画像化やLLM呼び出し）を安定稼働させる。
2. **ドキュメント画像レンダリングAPI (Document Render API)**:
   - PDFを各ページ高解像度のPNG画像にレンダリングする。
   - Word/ExcelをヘッドレスLibreOffice等を用いてPDF経由でPNG画像にレンダリングする。
3. **Visionテキスト抽出API (Vision Extraction API)**:
   - 画像リストを受け取り、Vision Modelを用いてレイアウトを維持した構造化テキストを抽出する。
4. **オントロジー生成API (Ontology Generation API)**:
   - 構造化テキストを受け取り、LLMを用いてJSONスキーマに従い、エンティティとリレーションを抽出する。
5. **型安全なバリデーション**:
   - LLMから返却されたJSONデータを Pydantic スキーマで検証し、ドメインモデルに変換する。
6. **DB非依存のオントロジー管理 (IGraphRepository)**:
   - 設定ファイル（`.env`）一つで複数のグラフDB（Neo4j、Apache AGE/PostgreSQL、Kùzu、またはインメモリRDF）に保存・同期する。
7. **GraphRAG 連携（エクスポート機能）**:
   - 外部グラフDBに直接ロードできる構造化JSONおよびTurtle形式（.ttl）ファイルでエクスポートする。
8. **マルチランゲージ対応 (Multi-language Support)**:
   - ontoNgn Console (Developer UI) および APIのエラーメッセージ表示は多言語（日本語・英語など）をサポート。
   - 抽出するオントロジーの記述言語設定をリクエストから動的に指定可能。
9. **ドキュメント収集エンジン (Ingestion Engine)**:
   - ユーザーからの手動アップロードだけでなく、ローカルフォルダの定期スキャンや、指定URLからの自動ダウンロードなど、多様な経路からのドキュメント収集と解析の自動トリガーをサポートする。
10. **グラフ可視化・探索機能 (Graph Visualization)**:
    - グラフDBに蓄積されたオントロジーデータを、動的なネットワークグラフとして可視化する機能。
    - キーワード検索を起点とし、ヒットしたノード周辺のサブグラフを段階的に探索・表示できる（vis-network等を利用）。

### 2.2 非機能要件 (Non-Functional Requirements)
1. **クリーンアーキテクチャによる低依存設計**:
   - データベースやLLM API、ファイルパーサーといった「技術的詳細」がビジネスロジックに影響を与えないよう、Pythonの抽象基盤クラス（`abc.ABC`）を用いた依存性注入（DI）を徹底する。
2. **非同期実行パフォーマンス**:
   - Pythonの非同期I/O（`async/await`）を活用し、画像の生成やLLM APIの呼び出しを非ブロッキングで並行処理する。
3. **データのポータビリティ**:
   - データの実体は標準的なトリプル（RDF）またはノード＆エッジ（LPG）として表現し、いつでもデータベース間を移行可能とする。
4. **英語ベースのソースコード規約 (English-based Codebase Standard)**:
   - プログラム内のコメント、Docstring、識別子（クラス名、関数名、変数名）はすべて英語（English）で記述・統一する。

### 2.3 アーキテクチャ設計方針
#### フロントエンド・ロジックレス（Thin Client / Dumb UI）
データの整合性保証、セキュリティ、およびオントロジー進化エージェントの推論結果の確実性を担保するため、「フロントエンド・ロジックレス（Dumb UI）」設計を徹底し、本体をバックグラウンドで自律的に動作するエンジンとして構築します。
- **ontoNgn Console (Minimal Dev UI)** は、純粋なプレゼンテーション表示層として動作し、APIから受け取った処理ステータス、ログ、AIエージェントからの「スキーマ進化の提案」を描画し、人間の「承認 / 却下」入力をそのままバックエンドAPIに中継する機能のみを担当します。
- クラス類似性の判定やコードのコンパイルなど、いかなるビジネスロジックもコンソール側には配置しません。

## 3. ソリューションのアーキテクチャ仕様（技術選定の背景）

各コア技術の選定背景の詳細については、[技術選定・比較検討報告書 (technology_decisions_report.md)](file:///workspaces/ontrogy/docs/technology_decisions_report.md) を参照してください。以下はその要約です。

### 3.1 Python (FastAPI) + HTMX の採用
- **LLMエコシステムとの親和性**: AI開発のデファクトであるPythonを採用することで、LangChain, LlamaIndexや各種PDF解析スクリプトをオーバーヘッドなしに直接呼び出せます。
- **ロジックレスフロントエンド**: フロントエンドにNext.js等の重厚なSPAフレームワークを使わず、**HTMX + Jinja2**を採用。FastAPIが返却するHTMLフラグメントをHTMXで差し替えるだけでSPA同等の速度とインタラクティブ性を実現します。これにより、クライアントとサーバー間の型共有の手間を排除しました。

### 3.2 データベース非依存設計 (`IGraphRepository`)
- **商用ライセンスリスクの回避**: Neo4j Community版はGPLv3ライセンスであり、自社ソースコード開示義務リスクがあります。DB非依存設計により、開発環境では `Neo4j`、商用/パッケージ配布環境では `Kùzu` (MIT) や `Apache AGE` (Apache 2.0) へ瞬時に切り替えられるようにしました。
- **インフラ環境への適応力**: PoCやローカル実行では組み込み型の `Kùzu` を使い、既存システムとの統合では PostgreSQL 拡張である `Apache AGE` を用いるなど、インフラに応じた適応力を高めます。

### 3.3 PostgreSQL をベースとしたマルチモデル統合価値
将来的な拡張として、PostgreSQLを単一のデータソース（Single Source of Truth）として機能させることができます。
- **RDB**: システム構成やユーザー管理。
- **Graph (Apache AGE)**: 手続き・アクター・書類の知識グラフ。
- **Vector (pgvector)**: チャンクテキストやノードのベクトル埋め込み。
- **Document (JSONB)**: 解析済みドキュメントのOCR構造データやログ。
これにより、データ同期遅延、スキーマ変更時の不整合、バックアップ運用の複雑さを最小化します。

## 4. ソリューションの使い方

### 4.1 開発環境の起動方法
本システムは FastAPI バックエンドエンジンおよび HTMX フロントエンドコンソールが一体となって動作します。

1. **依存関係のインストール**:
   ```bash
   pip install -r requirements.txt
   ```
   *(※ 内部で `pdf2image` を使用するため、システムに `poppler-utils` がインストールされている必要があります)*

2. **環境変数の設定 (`.env`)**:
   プロジェクトのルートディレクトリに `.env` ファイルを作成し、必要な設定を行います。
   ```env
   GRAPH_DB_TYPE=kuzu
   KUZU_DB_PATH=./data/kuzu
   LLM_API_BASE_URL=http://localhost:1234/v1
   VISION_MODEL_NAME=vision-model-name
   TEXT_MODEL_NAME=text-model-name
   ```

3. **サーバーの起動**:
   ```bash
   uvicorn app.main:app --reload --port 8000
   ```

4. **コンソールへのアクセス**:
   ブラウザで `http://localhost:8000/` にアクセスすると、`ontoNgn Console` 画面が表示されます。

### 4.2 基本的な処理フロー
1. **ドキュメントの収集 (Ingestion)**:
   Console UI の「新規アップロード」からの手動アップロードや、自動収集タスクによってドキュメントが登録されます。この時点のステータスは `Uploaded` となります。
2. **解析パイプラインの実行**:
   ドキュメントが登録されると、Workflow Orchestrator が非同期にパイプラインを開始し、処理の進捗に応じて状態を遷移させます。
   - **レンダリング**: ステータス `Rendering`。PDF/OfficeファイルをPNG画像群に変換します。
   - **テキスト化**: ステータス `Extracting_Text`。Vision Modelを用いて画像から構造化テキストを抽出します。
   - **オントロジー抽出**: ステータス `Generating_Ontology`。テキストからLLMを用いてJSON形式のオントロジーを抽出します。
3. **スキーマ進化の処理（必要時）**:
   ドキュメントから未知の概念が検出された場合、ステータスは `Pending_Evolution` となり処理が一時停止します。Console UIの「スキーマ進化管理」タブにAI Agentからの提案が表示され、開発者が **[承認]** または **[統合/却下]** を選択することでパイプラインが再開します。
4. **オントロジーの利用**:
   解析およびDBへの保存が全て完了すると、ステータスは `Completed` となります。万一処理が失敗した場合は `Failed` となり、UI上からリトライが可能です。完了した知識グラフデータはGraphRAG等でエクスポートして利用可能になります。
